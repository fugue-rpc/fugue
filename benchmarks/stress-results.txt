Platform: Windows 11, i7-11700F, Go 1.26
Date: 2026-04-24
Echo server: examples/echo-server (single process, no tuning)
Stress tool: examples/stress

Throughput ceiling ~65–70 k RPC/s on this machine. Primary bottleneck:
  - writeMu lock contention (40% of CPU, per pprof)
  - Windows CGO I/O overhead (25% of CPU; epoll on Linux would be ~3× cheaper)

Throughput scales roughly with 1/payload_size as expected for I/O-bound work.
512 KiB p99 spike (320 ms) is OS TCP send-buffer pressure — expected on loopback.
0 errors on all runs (end-of-duration context cancellations not counted).

── Key findings ──────────────────────────────────────────────────────────────

1. writeMu contention: with N streams per connection, N goroutines compete for
   the write mutex. Each stream holds the lock for one ws.Write syscall. At
   10+ streams/conn this is the dominant cost. Mitigation: fewer streams per
   connection OR a write queue (single writer goroutine per connection).

2. WebSocket read limit: coder/websocket defaults to 32 KiB. conn.go now calls
   ws.SetReadLimit(MaxPayloadSize + HeaderSize) = 4 MiB + 9 bytes, matching
   the protocol spec. Without this, any payload > 32 KiB was silently dropped.

3. Windows vs Linux: CGO overhead for I/O syscalls accounts for 25% of CPU
   on Windows. The same workload on Linux/epoll should be significantly faster.

4. Protocol correctness under concurrency: concurrent goroutines sharing one
   connection MUST send BEGIN frames in strictly ascending stream-ID order.
   The stress tool's original atomic-increment + separate lock approach violated
   this; the fix holds writeMu across both ID allocation and BEGIN write.
